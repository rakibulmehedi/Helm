# Validation Metrics

These metrics are used to evaluate the success of the Phase 8 MVP during founder testing and initial user rollouts. They measure behavioral reality, not stated intent.

## 1. Core Signals
- **Comprehension Signal:** User correctly explains what "Safe to Spend" means within 10 seconds of looking at the dashboard.
- **Trust Signal:** User explicitly states they would use the STS number to make a spending decision today.
- **Activation Signal:** User completes their first income entry (Received or Pending) without requiring verbal help from the founder.
- **Manual-Entry Friction Signal:** User actively complains about manual data entry OR user says manual entry is fine because the clarity is worth the effort.
- **Willingness-to-Pay Signal:** User provides a non-zero dollar amount when asked what they would pay for an automated version of this exact value proposition.
- **Retention Signal:** (For multi-day tests) User returns to the app unprompted on Day 2 or Day 3 to check their number.

## 2. Qualitative Flags
### Green Flags (Strong Product-Market Fit indicators)
- User asks: "Can I download this right now?"
- User immediately wants to enter their *real* data instead of dummy data.
- User says: "I always mix up my pending money with my real money."
- User actively expands the STS breakdown sheet to see the math.

### Red Flags (Pivot indicators)
- User asks: "Where is the pie chart for my expenses?" (Indicates they want a standard expense tracker, not an anxiety-reduction tool).
- User doesn't care about pending vs. received status.
- User says: "I already know what's in my bank account, why do I need this?" (Failure to convey the value of tax/buffer deductions).
- User gets stuck on the onboarding/settings flow.

## 3. 30-Day Validation Targets
If deployed to a beta group of 20 freelancers:
1. **70% Activation:** 14/20 add at least 3 income items in the first week.
2. **40% Weekly Active:** 8/20 check their Safe-to-Spend number at least twice a week.
3. **0 Critical Data Bugs:** No reports of STS calculating incorrectly or dropping data.
