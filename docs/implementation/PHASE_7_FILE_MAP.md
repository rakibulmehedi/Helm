## Expected Data Layer Files

### `data/datasources/income_local_data_source.dart`

Responsibility:
- Abstract interface defining CRUD contract
- Hive implementation: reads/writes to `incomeBox`
- Returns/accepts `IncomeModel` objects (not entities)
- No business logic — pure data access

Methods:
- `Future<void> addIncome(IncomeModel model)`
- `Future<void> updateIncome(IncomeModel model)`
- `Future<void> deleteIncome(String id)`
- `List<IncomeModel> getAll()`
- `Stream<BoxEvent>` or `ValueListenable` for reactive updates

---

### `data/models/income_model.dart`

Responsibility:
- Hive-serializable representation of an income entry
- `@HiveType(typeId: 2)` — FIXED, do NOT change
- All fields `@HiveField` annotated
- `toEntity()` method → returns `IncomeEntryEntity`
- `fromEntity()` factory constructor or static method
- Immutable fields with `final`

HiveField assignments:
| Field | HiveField index |
|-------|-----------------|
| `id` | 0 |
| `clientName` | 1 |
| `projectName` | 2 |
| `amount` | 3 |
| `currency` | 4 |
| `statusIndex` | 5 (stores `IncomeStatus.index`) |
| `expectedDate` | 6 |
| `receivedDate` | 7 |
| `notes` | 8 |
| `createdAt` | 9 |
| `updatedAt` | 10 |

DO NOT change field index assignments after first write.

---

### `data/models/income_model.g.dart`

- GENERATED by `dart run build_runner build`
- DO NOT edit manually
- DO NOT commit without the source `income_model.dart` change

---

### `data/repositories/income_repository_impl.dart`

Responsibility:
- Implements `IncomeRepository` (domain abstract class)
- Wraps `IncomeLocalDataSource`
- Maps `IncomeModel` → `IncomeEntryEntity` and back
- Error handling: wraps Hive errors, rethrows domain-safe exceptions

## Expected Domain Layer Files

### `domain/entities/income_entry_entity.dart`

Responsibility:
- Pure Dart — ZERO framework imports
- `IncomeStatus` enum defined here:
  ```dart
  enum IncomeStatus { expected, pending, received }
  ```
- `IncomeEntryEntity` class with all fields `final`
- `copyWith()` method for immutable updates
- No Hive annotations, no Flutter imports

Fields: id, clientName, projectName, amount, currency, status, expectedDate, receivedDate?, notes?, createdAt, updatedAt

---

### `domain/repositories/income_repository.dart`

Responsibility:
- Abstract class defining income data contract
- Methods match data source but return/accept entities, not models
- Methods:
  - `Future<void> addIncome(IncomeEntryEntity entity)`
  - `Future<void> updateIncome(IncomeEntryEntity entity)`
  - `Future<void> deleteIncome(String id)`
  - `List<IncomeEntryEntity> getAll()`
  - Reactive: `Stream` or `ValueListenable` for live updates

## Expected Presentation Layer Files

### `presentation/providers/income_providers.dart`

Defines all three income providers:

| Provider | Type | Scope |
|----------|------|-------|
| `incomeDataSourceProvider` | `Provider<IncomeLocalDataSource>` | Singleton |
| `incomeRepositoryProvider` | `Provider<IncomeRepository>` | Singleton |
| `incomeNotifierProvider` | `StateNotifierProvider<IncomeNotifier, List<IncomeEntryEntity>>` | App-level |

`IncomeNotifier` extends `StateNotifier<List<IncomeEntryEntity>>`.
Contains: `addIncome`, `updateIncome`, `deleteIncome`, `undoDelete` methods.
DO NOT define income providers in any transaction provider file.

---

### `presentation/views/income_list_screen.dart`

Responsibility:
- Route: `/income`
- Tab bar: All | Expected | Pending | Received
- Consumes `incomeNotifierProvider`
- Renders `IncomeCard` widgets
- Swipe-to-delete with undo SnackBar
- Renders appropriate `IncomeEmptyState` per tab
- Sort: by `expectedDate`, newest first within status group
- FAB or "+" button → navigates to `/add-income`
- Max 300 lines — extract widgets as needed

---

### `presentation/views/add_income_screen.dart`

Responsibility:
- Route: `/add-income`
- Form fields: clientName, projectName, amount, currency toggle (BDT/USD), expectedDate picker, notes (optional)
- Defaults: status = expected, currency = BDT, expectedDate = today
- Validation: per-field inline error messages
- On save: `incomeNotifierProvider.addIncome()` → navigate back → SnackBar "Income entry added"
- Double-submit guard: disable save button after tap until async completes
- `mounted` guard before navigation

---

### `presentation/views/edit_income_screen.dart`

Responsibility:
- Route: `/edit-income/:id`
- Pre-filled form from existing `IncomeEntryEntity`
- All fields editable including status (full status control in edit, not just quick-action)
- If entry not found by ID: show error state + back navigation ("Income entry not found.")
- On save: `incomeNotifierProvider.updateIncome()` → navigate back → SnackBar "Income entry updated"
- `mounted` guard before navigation
- Note: May be merged with `add_income_screen.dart` as a unified create/edit screen (same pattern as `AddTransactionScreen` if applicable)

---

### `presentation/views/widgets/income_card.dart`

Responsibility:
- Single income entry card
- Displays: clientName (headline), projectName (subtext), amount + currency, `IncomeStatusBadge`, expected/received date
- Quick-action button: "Mark Pending" (if expected) or "Mark Received" (if pending) — max 1 tap
- Expected status: no quick-action button to go backward (use edit screen)
- Received status: no forward quick-action (terminal state)
- Swipe-to-delete handler (calls notifier, triggers SnackBar in parent)

---

### `presentation/views/widgets/income_status_badge.dart`

Responsibility:
- Colored pill/chip showing status label
- Color mapping (ALL via AppColors — NO raw hex):
  - `expected` → soft grey
  - `pending` → soft blue or amber
  - `received` → gentle green
  - NO red for any status
- Displays human-readable label: "Expected" / "Pending" / "Received"

---

### `presentation/views/widgets/income_pipeline_summary.dart`

Responsibility:
- Three summary cards for dashboard
- Reads from `incomeNotifierProvider`
- Calculates: sum of `amount` where `status == expected` (by `expectedDate`, current month)
- Calculates: sum of `amount` where `status == pending` (by `expectedDate`, current month)
- Calculates: sum of `amount` where `status == received` (by `receivedDate`, current month)
- Visual hierarchy: Received most prominent, Expected least
- Tap on card: navigates to `/income` with appropriate status filter
- If all values zero: show dashboard empty state for income (not three zero cards)
- Works in light and dark mode

---

### `presentation/views/widgets/income_empty_state.dart`

Responsibility:
- Reusable empty state widget
- Parameters: `message` (String), `actionLabel` (String?), `onAction` (VoidCallback?)
- Used in: per-tab empty states in list screen, dashboard zero state

## Feature Boundary Enforcement

Income and Transactions are SEPARATE domains. They must never be merged.

| Concern | Transaction Feature | Income Feature |
|---------|--------------------|--------------------|
| Data model | `TransactionModel` (typeId: 0) | `IncomeModel` (typeId: 2) |
| Entity | `TransactionEntity` | `IncomeEntryEntity` |
| Repository (abstract) | `TransactionRepository` | `IncomeRepository` |
| Repository (impl) | `TransactionRepositoryImpl` | `IncomeRepositoryImpl` |
| Data source | `TransactionLocalDataSource` | `IncomeLocalDataSource` |
| Hive box | `transactionBox` | `incomeBox` |
| State notifier | `TransactionsNotifier` | `IncomeNotifier` |
| Provider | `transactionsProvider` | `incomeNotifierProvider` |
| Screens | `/add-transaction`, etc. | `/income`, `/add-income`, `/edit-income/:id` |
| Feature folder | `lib/features/transactions/` | `lib/features/income/` |

EXPLICITLY FORBIDDEN:
- Shared `FinancialEntry` base class merging both domains
- Income providers that `ref.watch` transaction providers (or vice versa)
- Income entries stored in `transactionBox`
- Income entries rendered in transaction list
- Income fields added to `TransactionEntity` or `TransactionModel`
